# visualization

## Overview

`visualization` is a visualization utility component for ROS 2 scenarios. It is mainly used to provide basic visualization capabilities for robotic systems, including:

- Launching RViz for navigation, SLAM, RGBD, and 3D LiDAR visualization
- Providing a WebSocket-based image streaming node that forwards compressed image topics from ROS to a browser
- Working with frontend HTML templates to offer a lightweight web-based image viewing capability

This component mainly addresses the following needs:

- Quickly viewing sensors, maps, navigation status, and other visualization data during robot debugging
- Accessing image results through a browser in headless or remote access scenarios
- Providing a unified visualization entry for development, integration, debugging, and demos

## Features

Currently supported:

- Launching RViz visualization for different scenarios via launch files
  - `display_navigation.launch.py`
  - `display_slam.launch.py`
  - `display_rgbd.launch.py`
  - `display_rtab_3dlidar.launch.py`
- Launching the `websocket_cpp_node`
- Subscribing to ROS 2 compressed image topics and forwarding image data to web clients through WebSocket
- Accessing visualization results through template pages
- Configuring image topics and WebSocket ports through launch parameters

Currently unsupported or requiring attention:

- The current web page capability is focused on image display only and is not a full robot operation/monitoring platform
- The WebSocket node currently forwards compressed image data and does not include production-grade security features such as authentication, encryption, or access control by default
- Resource directories and package names referenced by different launch files should be verified against the actual installed contents, such as RViz config files and upper-level visualization resource packages

## Quick Start

### Environment Preparation

Please make sure the following prerequisites are ready:

- ROS 2 is installed and configured
- The current workspace dependencies and source code are fully synced
- Basic build tools such as `colcon` and `ament` are available
- If you use the web image streaming feature, the network should be reachable and the device IP should be accessible from a browser

### Build Inside the SDK

This repository provides a unified SDK build system. It is recommended to use the SDK build flow instead of running `colcon build` manually whenever possible.

The standard workflow is:

1. Load the build environment script
2. Select a target configuration via `lunch`
3. Build using `m`, `mm`, or `build/build.sh`

Run the following in the repository root:

```bash
source build/envsetup.sh
```

Select a build target, for example:

```bash
lunch
# Or specify directly
lunch k3-com260-minimal
```

If the current target configuration already enables the ROS 2 packages including `visualization`, you can directly build all ROS 2 components:

```bash
m -R
```

Or use the equivalent command:

```bash
./build/build.sh ros2
```

If you only want to build this component, the recommended way is to use the single-package build flow:

```bash
source build/envsetup.sh
lunch k3-com260-minimal

cd middleware/ros2/tools/visualization
mm
```

You can also run it directly from the repository root:

```bash
source build/envsetup.sh
lunch k3-com260-minimal

./build/build.sh package middleware/ros2/tools/visualization
```

Build outputs are generated by default under:

- `output/staging`: install directory for development and debugging
- `output/rootfs`: root filesystem directory for deployment

After building, if you want to run ROS 2 nodes, it is recommended to load the ROS 2 runtime environment provided by the SDK:

```bash
source build/envsetup.sh
sros2_setup
```

Additional notes:

- `m -R` / `./build/build.sh ros2` builds ROS 2 packages enabled by the current `target/*.json` configuration
- `mm` is suitable for fast iteration inside a single component directory
- If `visualization` is not included in the current target configuration, you need to check and update the corresponding `target/*.json`

### Standalone Build

If you want to build this package independently, go to the workspace root and load the ROS 2 environment first:

```bash
source /opt/ros/<ros2_distro>/setup.bash
```

Then run:

```bash
colcon build --packages-select visualization
```

After the build is complete, run:

```bash
source install/setup.bash
```

### Run Examples

#### 1. Start the WebSocket Image Streaming Node

```bash
ros2 launch visualization websocket_cpp.launch.py
```

To specify the image topic and port:

```bash
ros2 launch visualization websocket_cpp.launch.py image_topic:=/result_img port:=8080
```

After startup, access the page in your browser using the device IP and port, for example:

```text
http://<device-ip>:8080
```

#### 2. Start RViz Visualization

For example, to start the navigation scenario:

```bash
ros2 launch visualization display_navigation.launch.py
```

Other scenarios can be launched as needed:

```bash
ros2 launch visualization display_slam.launch.py
ros2 launch visualization display_rgbd.launch.py
ros2 launch visualization display_rtab_3dlidar.launch.py
```

## FAQ

### 1. What should I do if the browser page cannot be opened?

Please check the following in order:

- Whether `websocket_cpp_node` is running correctly
- Whether the port is already occupied
- Whether the device IP is correct
- Whether the host firewall allows the target port
- Whether the browser URL is `http://<device-ip>:<port>`

### 2. What should I do if the page opens but no image is shown?

Please check:

- Whether the `image_topic` parameter matches the actual published compressed image topic
- Whether the upstream node is publishing `sensor_msgs/msg/CompressedImage`
- Whether the image data is being updated normally
- Whether the browser can reach the device running the node

### 3. What should I do if RViz fails to start or does not show the expected content?

Please check:

- Whether the corresponding RViz config file exists
- Whether the required topics, TF, map, or sensor data are being published correctly
- Whether the package names and resource directories referenced in the launch files match the actual workspace contents
- Whether you have correctly run `source install/setup.bash`

### 4. What should I do if the build fails?

Please check:

- Whether ROS 2 dependencies are installed completely
- Whether dependencies such as OpenCV and Boost are available
- Whether there are upstream packages in the current workspace that failed to build
- Whether you are using the correct build environment and toolchain
